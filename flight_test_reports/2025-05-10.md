## Motor: I211W
## Apogee: 380m / 1246ft
## Max velocity: 84.19 m/s or 303.084 km/h

# What worked nominally
- Launch detection
- Burnout detection
- Main deployment detection
- Landing detection
- Logging and radio

# What did not work
- Apogee detection was too early
- Barometric filtering was too aggressive

## Why this happened:
### 2 faulty apogee detection conditions were true:
- GNSS NED vertical velocity perceived upwards velocity as negative velocity, which triggered 1/2 apogee conditions
- Gyro tilt angle did not take mounting orientation into account, so roll angle was perceived as tilt angle, easily exceeding 90 degrees and triggering the second apogee condition

### Baro filtering:
- 2D Kalman filter with low process covariance did not react to fast ascent in time
- Baro altitude climbed up slower than expected
- Baro velocity was even less responsive and lagged behind even further

# Fixes/improvements:
## Sensor logic
1. Multiply GNSS NED vertical velocity by `-1` to correct readings
2. Rotate gyro readings the same way as accelerometer readings to account for any orientation
3. Remove IMU filtering, 5 second calibration refresh is sufficient to deal with drift and integration error accumulation
4. Redo barometric filtering, 2D kalman updates way too slowly

## Flight logic
- Reduce IMU persistence checks
- Add flight time tracking
- Add a false launch detect system - between 0.25 and 1 second after launch, check if acceleration falls below threshold and `!canDetectBurnout` (rocket has not exceeded 20m/s at any point), return to ARMED mode 
- Landing detection should not use accelerometer

## Radio
- Transmit messages as binary instead of string

## Logging
- Log using a timer instead of whenever new sensor data is received for better postprocessing experience
- Log seconds instead of milliseconds
